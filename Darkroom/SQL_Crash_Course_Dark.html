1. What is SQL?

📂 Database: photography_db

SQL stands for Structured Query Language. It is the language you use to talk to a database — to ask questions, retrieve data, add new records, or make changes.

Our database is called photography_db. It contains photographers from 20 countries, their shoots around the world, and the equipment they own. This is the same data used in the Python course — two languages, one dataset.

Key TermsA database is a collection of organised data. A table is one set of related data inside it. A row is one record. A column is one piece of information about that record.

2. SELECT — Reading Data

📂 Table: photographers

The most important SQL command is SELECT. You will use it in almost every query you ever write. It tells the database which columns you want to see.

-- Get all columns from the photographers table SELECT * FROM photographers;
-- Get only name and country SELECT name, country FROM photographers;
Result
namecountry
SatdivaUSA
Carlos MendezBrazil
Priya NairIndia
James OkonkwoNigeria
-- Rename a column in the output using AS SELECT name AS photographer, specialty AS type_of_work, rating FROM photographers;
Use * sparingly. SELECT * returns every column. It is fine for exploring, but in real queries list only the columns you actually need.

3. WHERE — Filtering

📂 Table: shoots

WHERE filters your results. Instead of every row, you get only the rows that match your condition. Think of it as the question after "show me only...".

-- Only shoots in Florida SELECT event, location, year FROM shoots WHERE location = 'Florida';
-- Only shoots after 2020 SELECT event, year, photos_taken FROM shoots WHERE year > 2020;
-- Combine two conditions with AND SELECT event, location, year FROM shoots WHERE location = 'Florida' AND year > 2020;
Result
eventlocationyear
Wellington PoloFlorida2022
Grand Prix WEFFlorida2023
GCDA ClassicFlorida2024
Operators you can use= (equals)  ·  != (not equal)  ·  > < >= <= (comparisons)  ·  AND / OR (combine)  ·  LIKE (pattern match)  ·  IN (list of values)
-- LIKE — partial match (% is a wildcard) SELECT name, country FROM photographers WHERE name LIKE 'S%'; -- IN — match any value from a list SELECT name, country FROM photographers WHERE country IN ('USA', 'Brazil', 'Nigeria');
Text values need quotes. Numbers do not. WHERE location = 'Florida' — WHERE year > 2020. Mix them up and you get an error.

4. ORDER BY — Sorting

📂 Table: shoots

ORDER BY sorts your results. By year, by name, by number of shots. Ascending (A→Z, oldest→newest) or descending (newest→oldest, highest→lowest).

-- Most recent shoots first SELECT event, location, year FROM shoots ORDER BY year DESC;
-- Shoots with the most photos taken, highest first SELECT event, photos_taken FROM shoots ORDER BY photos_taken DESC;
-- Sort by two columns — year first, then event name SELECT event, location, year FROM shoots ORDER BY year DESC, event ASC;
Result
eventlocationyear
GCDA ClassicFlorida2024
Grand Prix WEFFlorida2024
Wellington PoloFlorida2023
Dressage CupAachen2023
ASC = ascending (default, smallest to largest, A to Z). DESC = descending (largest to smallest, Z to A). When in doubt, add DESC to see the most recent or highest values first.

5. LIMIT & DISTINCT

📂 Table: shoots · photographers

Two small commands that you will use constantly. LIMIT controls how many rows you get back. DISTINCT removes duplicate values.

LIMIT

-- Show only the 5 most recent shoots SELECT event, location, year FROM shoots ORDER BY year DESC LIMIT 5;
-- The single shoot with the most photos taken SELECT event, photos_taken FROM shoots ORDER BY photos_taken DESC LIMIT 1;

DISTINCT

-- Which countries are represented? (no duplicates) SELECT DISTINCT country FROM photographers ORDER BY country;
-- Which locations have been used for shoots? SELECT DISTINCT location FROM shoots ORDER BY location;
Result
location
Aachen
Florida
Geneva
London
Wellington
Always use LIMIT when exploring a large table. 750,000 rows is a lot to look at. LIMIT 10 keeps the database fast and your screen readable.

6. COUNT, AVG, MIN, MAX

📂 Table: shoots · photographers

Aggregate functions calculate values across multiple rows. Instead of seeing individual rows, you get a single summary number. Essential for understanding your archive at scale.

-- How many shoots in the database? SELECT COUNT(*) AS total_shoots FROM shoots; -- Average photos taken per shoot SELECT AVG(photos_taken) AS avg_photos FROM shoots; -- Highest and lowest number of photos in a single shoot SELECT MAX(photos_taken) AS biggest_shoot, MIN(photos_taken) AS smallest_shoot FROM shoots;
Result
total_shootsavg_photosbiggest_shootsmallest_shoot
750,0008472,400120
-- Total photos exported across all shoots SELECT SUM(photos_exported) AS total_exported FROM shoots; -- How many photographers are rated above 4.0? SELECT COUNT(*) AS top_rated FROM photographers WHERE rating > 4.0;
SUM · AVG · MAX · MIN · COUNT — these five functions answer most real-world questions about your archive. How much? How many? What is the best? What is the worst?

7. GROUP BY & HAVING

📂 Table: shoots · photographers

GROUP BY groups rows together so you can count, sum or average per category. Instead of "how many shoots total" you get "how many shoots per year" or "per location".

-- How many shoots per year? SELECT year, COUNT(*) AS shoots_that_year FROM shoots GROUP BY year ORDER BY year DESC;
Result
yearshoots_that_year
2024312
2023298
2022341
2021276
-- Average photos per shoot, grouped by location SELECT location, COUNT(*) AS total_shoots, ROUND(AVG(photos_taken), 0) AS avg_photos FROM shoots GROUP BY location ORDER BY total_shoots DESC;

HAVING — filter after grouping

WHERE filters rows before grouping. HAVING filters groups after grouping. Use HAVING when your condition involves an aggregate function.

-- Only locations with more than 50 shoots SELECT location, COUNT(*) AS shoot_count FROM shoots GROUP BY location HAVING COUNT(*) > 50 ORDER BY shoot_count DESC;
The ruleEvery column in SELECT that is not inside a function (COUNT, SUM, AVG) must appear in GROUP BY. If you SELECT it, GROUP BY it.
WHERE vs HAVING: WHERE filters rows before the group is formed. HAVING filters the groups after they are formed. When in doubt — if your condition uses COUNT, SUM, AVG — use HAVING.

8. JOIN — Two Tables

📂 Tables: photographers · shoots

JOIN connects two tables together using a shared column. This is where SQL becomes truly powerful — linking a photographer's name to all their shoots, across two separate tables.

-- Connect shoots to photographer names SELECT photographers.name, shoots.event, shoots.location, shoots.year FROM shoots JOIN photographers ON shoots.photographer_id = photographers.photographer_id ORDER BY shoots.year DESC LIMIT 5;
Result
nameeventlocationyear
SatdivaGCDA ClassicFlorida2024
SatdivaGrand Prix WEFFlorida2024
Priya NairMumbai Portrait WeekIndia2023
James OkonkwoLagos DocumentaryNigeria2023
-- Use table aliases to keep queries shorter SELECT p.name, s.event, s.year, s.photos_taken FROM shoots s JOIN photographers p ON s.photographer_id = p.photographer_id WHERE s.year >= 2022 ORDER BY s.photos_taken DESC;
The shared keyshoots.photographer_id matches photographers.photographer_id. That shared column is what connects the two tables. Without it, the JOIN has nothing to match on.
Table aliases — give each table a short letter (p for photographers, s for shoots). It makes long queries much easier to read and write.

9. Multiple JOINs

📂 Tables: photographers · shoots · equipment

You can JOIN more than two tables in a single query. Each JOIN adds one more connection. The key is always a shared column between two tables.

-- Photographers + their shoots + the equipment used SELECT p.name AS photographer, s.event, s.year, e.brand || ' ' || e.model AS camera FROM shoots s JOIN photographers p ON s.photographer_id = p.photographer_id JOIN equipment e ON e.photographer_id = p.photographer_id WHERE e.type = 'camera_body' ORDER BY s.year DESC LIMIT 10;
Result
photographereventyearcamera
SatdivaGCDA Classic2024Canon EOS R5
SatdivaGrand Prix WEF2024Canon EOS R5
Carlos MendezRio Street Series2023Sony A7 IV
-- How many pieces of equipment does each photographer own? SELECT p.name, COUNT(e.item_id) AS gear_count, SUM(e.purchase_price_usd) AS total_invested FROM photographers p JOIN equipment e ON e.photographer_id = p.photographer_id GROUP BY p.name ORDER BY total_invested DESC;
Each JOIN is one more bridge. First bridge: shoots → photographers. Second bridge: equipment → photographers. The order matters — each new table must share a column with one already in the query.

10. Subqueries

📂 Tables: photographers · shoots

A subquery is a query inside a query. It lets you use the result of one question as the input for another. The inner query runs first and passes its result to the outer query.

-- Find all shoots above the average photo count SELECT event, location, photos_taken FROM shoots WHERE photos_taken > ( SELECT AVG(photos_taken) FROM shoots ) ORDER BY photos_taken DESC;
-- Photographers who have done more than 50 shoots SELECT name, country, specialty FROM photographers WHERE photographer_id IN ( SELECT photographer_id FROM shoots GROUP BY photographer_id HAVING COUNT(*) > 50 );
Result
namecountryspecialty
SatdivaUSAEquestrian
Carlos MendezBrazilStreet
James OkonkwoNigeriaDocumentary
The inner query runs firstIts result is passed to the outer query. Think of it as answering a smaller question in order to answer the bigger one. The inner query always goes inside parentheses.

11. INSERT & UPDATE

📂 Tables: photographers · shoots

Everything so far has been reading data. INSERT adds new rows. UPDATE changes existing ones. These are the commands you use to keep your archive current.

INSERT — adding a new record

-- Add a new shoot to the database INSERT INTO shoots (photographer_id, event, location, year, photos_taken, photos_exported) VALUES (1, 'WEF Grand Prix', 'Florida', 2025, 1340, 98);
-- Add a new photographer INSERT INTO photographers (name, country, specialty, years_experience, rating) VALUES ('Amara Diallo', 'Senegal', 'Wildlife', 8, 4.3);

UPDATE — changing existing data

-- Update the rating for one photographer UPDATE photographers SET rating = 4.8 WHERE photographer_id = 1; -- Update photos_exported for a specific shoot UPDATE shoots SET photos_exported = 112 WHERE shoot_id = 4471;
Always use WHERE with UPDATE. Without WHERE, the UPDATE applies to every row in the table. That is almost never what you want. Check your WHERE condition with a SELECT first before running an UPDATE.
The full SQL toolkitSELECT · WHERE · ORDER BY · LIMIT · DISTINCT · COUNT · AVG · SUM · MIN · MAX · GROUP BY · HAVING · JOIN · Subqueries · INSERT · UPDATE. That is everything you need to query, understand and maintain your entire photography archive.
