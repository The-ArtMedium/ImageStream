How to think before you type

You already know
how to query

Before you learn SQL syntax, understand this: you have been querying data your entire life. Every time you asked a question about a group of things, you were doing what a database does.

You think
"Show me all the portrait photographers with a rating above 4.5"
SELECT name, rating
FROM photographers
WHERE specialty = 'Portrait'
  AND rating > 4.5;
You think
"Who are the top 3 earners this year, sorted by total fees?"
SELECT name, SUM(fee_usd)
FROM shoots
GROUP BY name
ORDER BY fee_usd DESC
LIMIT 3;
You think
"What countries have more than two photographers in the database?"
SELECT country, COUNT(*)
FROM photographers
GROUP BY country
HAVING COUNT(*) > 2;
You think
"Give me all shoots in Lagos or Nairobi sorted by most recent"
SELECT * FROM shoots
WHERE location IN ('Lagos', 'Nairobi')
ORDER BY year DESC;
The shift in thinking

SQL does not require you to think like a programmer. It requires you to think like someone asking a clear question about organised information. If you can form the question in plain language first, the SQL almost writes itself.

The most important diagram in SQL

The order you write it
is not the order it runs

This single misunderstanding causes more confusion than any other in SQL. You write a query in one order. The database runs it in a completely different order. Once you see this, error messages start to make sense.

The order you write it
1
SELECT
what columns
2
FROM
which table
3
WHERE
filter rows
4
GROUP BY
group rows
5
HAVING
filter groups
6
ORDER BY
sort results
7
LIMIT
how many rows
⟹
The order the database runs it
1
FROM
go get the table first
2
WHERE
remove rows that don't match
3
GROUP BY
form the groups
4
HAVING
remove groups that don't match
5
SELECT
now pick the columns
6
ORDER BY
sort what remains
7
LIMIT
cut to the right number
Why this matters — three things it explains immediately
  • Why you cannot use a SELECT alias inside WHERE — SELECT hasn't run yet when WHERE executes
  • Why WHERE cannot use COUNT() or AVG() — those are calculated in SELECT, which runs later
  • Why HAVING exists at all — it runs after GROUP BY, so it can filter using aggregate results that WHERE never sees
A question worth asking

Can HAVING exist
without GROUP BY?

Yes. And understanding why reveals that you truly understand what GROUP BY does — not just where to type it.

What happens without GROUP BY

When there is no GROUP BY, the entire table is treated as one single group. Every row belongs to that one group. So HAVING applies to the whole table as if it were one.

This query is technically valid:

SELECT COUNT(*) AS total
FROM photographers
HAVING COUNT(*) > 5;

It asks: does the entire photographers table have more than 5 rows? If yes, return the count. If no, return nothing. Almost never what you actually want — but it works, and knowing why it works means you understand the foundation.

Technically valid
HAVING without GROUP BY treats the whole table as one group. The query will run without error.
Almost never useful
If you find yourself writing HAVING without GROUP BY, you almost certainly want WHERE instead — or you need to add a GROUP BY.
The foundation test

If someone asks you "can HAVING exist without GROUP BY?" and you can answer it clearly — you understand the execution order. That means WHERE, HAVING, GROUP BY, and aggregate functions will never confuse you again. That is the foundation. Everything else is just syntax.

Two languages, one goal

SQL and Python —
not rivals, partners

You will learn both in this series. They are not competing. They are tools that work together, and most data analysts use both every day.

LanguageWhat it doesWhen you use it
SQLSpeaks directly to databases — asks questions, retrieves, filters, groups data stored in tablesWhen your data lives in a database and you need to extract or summarise it
PythonA general programming language — can do almost anything, including load files, analyse data, and draw chartsWhen you have the data and want to reshape, calculate, visualise, or automate things with it
The real world workflow

A data analyst typically uses SQL to ask "give me all shoots in 2023 with fee above 500" — pulling the data out of the database. Then uses Python with pandas to clean it, calculate averages, and matplotlib to turn it into a chart. SQL extracts. Python analyses and visualises.
